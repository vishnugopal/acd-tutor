acd-tutor — how it's built, where it creaks,
and how to restructure it

A senior-engineer read of the codebase: an agentic tutor on the Flue framework with two frontends (Ink console + React web). The bones are good — clear layering, small modules, real tests. Three systemic issues make change harder than it should be: agent identity scattered as strings across ~7 files, the same transport types defined three times (with one genuine layering violation), and one 315-line god component hiding two real save bugs — one of which can silently lose a student's work.

1 · System overview

Three compilation worlds, two processes, one filesystem seam

The most load-bearing fact about this codebase: source files live in three different compilation worlds. Some are bundled by flue build into the agent subprocess, some run on the Bun host, some are bundled for the browser — and a few (tools.ts, workspaces.ts, open-request.ts) straddle worlds. Every refactoring decision below respects these boundaries.

Flue build (never importable from host) Host process Browser bundle Filesystem contract

Key rule the refactor must preserve: profiles import SKILL.md via with { type: "skill" }, so they only resolve inside Flue's build — host code must never import them. And anything shared with the browser must be free of node:* and process.env. That constrains where a "shared" layer can live and what it may contain: pure data and calculations only — which happens to be exactly what the project's own style guide asks for.

3 · The cost of change today

Adding one agent touches 6 files — with zero compile-time checks

Agent identity is a string repeated across the codebase, and nothing verifies the copies agree. Three of these files are demanded by Flue and will always exist (green). The other three are accidental duplication (numbered red) — each one a silent failure if forgotten.

1

src/agents/<id>.ts

The Flue agent stub — 16 lines, 94% identical to its siblings. Filename is the agent id.

Required by Flue. One agent per top-level file is load-bearing — keep it.
2

src/agents/profiles/<id>.ts

Instructions + SKILL.md import + tools. Only resolves inside flue build.

Required by Flue. Stays.
3

src/skills/<id>/SKILL.md

The tutor's actual pedagogy. Where behavior changes belong.

Required by Flue. Stays.
4

profiles/registry.ts:15-58

Hardcoded AGENT_CHOICES: id, label, greeting, actions.

Forget it: agent is invisible in both the console menu and the web home screen. No error.
5

workspaces.ts:19-22

A second map keyed by the same id strings → scratch dir.

Forget it: file APIs 404, lesson tools break, IDE screen stays empty. Silent at build time.
6

web/client/data/agents.ts:22-61

A third id-keyed map: editor kind, icon, and a hardcoded 7-step course outline.

Forget it: web client has no UI config for the agent — undefined lookups at runtime.

After the refactor: steps 4–6 collapse into one entry in src/shared/catalog.ts, where a Record<AgentId, AgentDefinition> makes the compiler refuse to build until the entry exists — and a filesystem-consistency unit test fails bun test with a named error if the stub, profile, or skill directory is missing or orphaned.

4 · Type duplication

One concept, three definitions, zero shared source

The streaming chunk — the single most important type in the system — is defined three times, structurally identical. The console even has a no-op converter (console/App.tsx:13-15) translating the type to itself. Agent metadata has the same disease: AgentChoice vs AgentInfo, ConsoleAction vs AgentAction.

The layering violation it causes

Because the canonical AgentChoice lives in src/console/types.ts, both the registry (registry.ts:1) and the web server (web/server.ts:3) import from the console frontend. A change to console UI types can break the web API surface. The fix is mechanical once shared/catalog.ts exists: console and web both alias the shared AgentPresentation.

5 · Streaming pipeline

Browser → web server → agent-io → Flue, with four soft spots

The happy path works well — SSE end to end, with a keepalive heartbeat. The soft spots are all at the edges: an unguarded parser, no way to cancel, and a stream that ends without done being indistinguishable from a complete reply.

① Unguarded parse. api.ts:75 calls JSON.parse on each SSE frame with no guard — one malformed frame crashes the stream handler. Fix: parseStreamFrame() returns null for garbage.
② No cancellation. No AbortController anywhere — navigating away or resetting chat leaves the fetch streaming in the background (useAgentChat.ts).
③ Implicit done. If the connection drops before the done frame, the client renders a truncated reply as if complete. Fix: reader-ends-without-done → throw.
④ Unbounded maps. Acceptable for localhost, but a 6-line size cap documents the intent.
6 · The IdeLessonScreen save machine

Two real bugs hiding in an implicit state machine

IdeLessonScreen.tsx (315 lines) interleaves autosave, file switching, tutor-driven file sync, open-request polling, chat, and toasts — with the save lifecycle spread across useState, three refs, and a debounced effect. Drawn as the state machine it implicitly is, the two bugs become visible.

The fix is structural, not a patch: extract the lifecycle into a pure reducer (lib/autosave.ts — nextSaveState(state, event), trivially unit-testable), wrap it in a thin useAutosave hook whose flush() returns false on failure (callers decide whether to proceed), and re-arm the timer once on error. The file-sync decision tree (open-request wins → first file → newer-and-clean → tutor-rewrote-current, lines 96-144) gets the same treatment: a pure planSync() calculation with tests pinning today's behavior, then a useLessonFileSync hook. The screen drops to ~120 lines of layout.

8 · Target architecture

One new layer fixes the two structural problems

A layer 0: src/shared/ — browser-safe, Flue-build-safe, dependency-free pure data and calculations. Everything above imports it; nothing in it imports anything. Toggle to compare.

The catalog: one entry per agent, compiler-enforced

// src/shared/catalog.ts — data + calculations only
export type AgentId =
  | "acd-tutor"
  | "argumentative-essay-tutor"
  | "socratic-tutor";

export interface AgentDefinition {
  id: AgentId;
  label: string;
  description: string;
  greeting: string;
  farewell?: string;
  actions: AgentAction[];
  /** Present iff the agent manages lesson files. */
  workspace?: {
    dirEnvVar: string;     // "ACD_TUTOR_SCRATCH_DIR"
    defaultDir: string;    // "/tmp/acd-tutor/scratch"
    editor: "code" | "markdown";
  };
  /** Dashboard stepper; absent = no course card. */
  course?: CourseOutline;
}

export const AGENTS = {
  "acd-tutor": { /* … */ },
  // forget an entry → compile error
} satisfies Record<AgentId, AgentDefinition>;

export function isAgentId(v: string): v is AgentId;
export function toPresentation(d: AgentDefinition):
  AgentPresentation;   // the wire/menu subset
// src/shared/chunks.ts
export type AgentChunk =
  | { kind: "text";  text: string }
  | { kind: "debug"; text: string };

export type StreamFrame =
  | AgentChunk
  | { kind: "done" }
  | { kind: "error"; text: string };

/** null for malformed frames — skip, don't crash. */
export function parseStreamFrame(raw: string):
  StreamFrame | null;

What derives from the catalog: AGENT_CHOICES becomes AGENT_LIST.map(toPresentation); WORKSPACES becomes a host-side workspaceDir(id) resolving env over defaultDir; the web client's data/agents.ts shrinks to a cosmetic Record<AgentId, {icon, iconBg}> — still compile-checked. The unavoidable Flue trio (stub, profile, skill) is guarded by a filesystem-consistency test: readdir both directories and assert every AgentId has its stub, profile, and skill — and no orphans exist.

One subtlety kept on purpose: console and web server continue to receive their agent list by injection (from main.ts / web/main.ts) rather than importing the catalog — the e2e suite injects a tutor-faux agent that isn't in the catalog.

9 · Roadmap

Four phases, each independently shippable

Each phase ends green: bunx tsc --noEmit + bun test, plus bun run test:e2e whenever the agent or streaming path is touched (per AGENTS.md). Foundation first — everything else builds on the shared layer.

Phase 1 — Shared foundation fixes #1 #2

+ shared/catalog.ts + shared/chunks.ts + shared/lesson-names.ts workspaces.ts registry.ts → derived console/types.ts console/App.tsx web/server.ts web/client/types.ts data/agents.ts → cosmetic AGENTS.md

Tests: catalog test incl. the filesystem-consistency check (AgentId ↔ stub/profile/skill, no orphans); parseStreamFrame (valid / garbage / wrong-shape → null); moved lesson-name tests. Risk: shared/ must stay free of node:* and process.env or the browser bundle and flue build fail in different ways — both compilers are the guard. Keep agent lists injected (e2e faux agent isn't in the catalog).

Phase 2 — One lesson-file module fixes #6 #10 #13 #14

+ lesson-files.ts tools.ts → sentinel adapter web/server.ts web/files.ts

Tests: new lesson-files suite (traversal: ../x, /etc/x; dotfiles; open-request round-trip incl. malformed JSON); tools.test.ts slims to sentinel mapping + spawn arg shape. Re-run e2e. Risk: the $EDITOR spawn change (naive split, args-array) — verify manually with EDITOR="zed -w". Tool descriptions/sentinels must not change: the model prompt depends on them.

Phase 3 — Streaming hardening fixes #7 #8 #9 #11

web/server.ts (StreamFrame · readJsonBody · session cap) web/client/api.ts (guarded parse · AbortSignal · explicit done) useAgentChat.ts (AbortController) + console/useChatStream.ts (pure foldChunk) console/App.tsx

Tests: mocked-fetch stream tests (happy path / malformed frame skipped / missing done throws / abort); foldChunk unit tests; server body-validation tests via startWebServer on port 0. Risk: missing-done now surfaces as an error — confirm useAgentChat shows the error bubble, not a truncated reply. Heartbeat comments must still be filtered by the data:  prefix check.

Phase 4 — Web client decomposition fixes #3 #4 #5 #12

+ client/lib/autosave.ts (reducer) + client/hooks/useAutosave.ts + client/hooks/useLessonFileSync.ts (pure planSync) + client/components/LessonScreenShell.tsx IdeLessonScreen.tsx → ~120 lines ChatLessonScreen.tsx

Tests: autosave reducer (edit→unsaved; save-ok-with-concurrent-edit→unsaved; save-fail→error+retry); planSync decision table pinning today's behavior before touching the hook. Fixes both save bugs via flush() returning false on failure. Risk: the sync decision tree is subtle — extract pure, pin with tests, then rewire. Manual browser pass through both tutors.

Phase 5 (optional) — Test debt

+ console InputBar/Transcript/Menu tests e2e: per-test session ids, adaptive readiness polling

Pure test work, no production code. Do it when it starts hurting, not before.

10 · Anti-goals

What we deliberately won't do

For a ~580-LOC backend, the failure mode isn't under-engineering — it's the opposite. These were considered and rejected.

Codegen for agent stubs

Flue's one-agent-per-file discovery is load-bearing. The filesystem-consistency test is the right-sized guard, not a generator.

Validation library (zod etc.)

Three request bodies. A 10-line readJsonBody(req, guard) helper suffices. Bun-only / no-new-deps holds.

Real session eviction (LRU, TTLs)

Localhost, one learner. A size cap plus a comment documenting the decision is proportionate.

Unifying console and web UIs

They share types and transport, not rendering. A "ChatEngine" abstraction over both send loops would couple things that differ legitimately (queueing, debug display).

Serving UI config over /api/agents

The client is bundled from the same repo — importing the catalog is simpler than versioning a wire schema for icons.

Building gamification

XP/streak/hearts stay labeled placeholders. Phase 1 deletes the dead course-steps copy in gamification.ts; the rest waits for a real product decision.

Monorepo / publishing shared/ as a package

A directory with import discipline — enforced by the browser bundle and flue build both compiling it — is enough at this size.

Generalizing LessonFileStore

Two consumers. No pluggable backends, no event emitters. Two call sites don't justify an abstraction.
